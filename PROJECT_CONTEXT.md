# Flexodoro Project Context

Read this file first in future sessions. It is meant to be enough context to understand the project before making changes. Per `AGENTS.md`, this repo uses a modern Next.js version with possible breaking changes, so consult `node_modules/next/dist/docs/` before framework-level edits.

## What This Project Is

Flexodoro is a focus timer product built with Next.js App Router, TypeScript, Tailwind CSS, Zustand, Prisma, and PostgreSQL. It supports fixed Pomodoro sessions and flexible deep-work sessions with adaptive break suggestions, generated focus audio, theme preferences, and analytics APIs.

The codebase currently contains both polished Figma-derived screens and more production-like feature modules. This split is the most important architectural fact in the repo.

## Timer Behavior

Production timer behavior is implemented in `features/timer/hooks/use-timer-engine.ts`.

- Runs only on the client.
- Stores a target end timestamp and computes remaining time from wall clock, avoiding interval drift.
- Ticks every 250ms.
- On work completion:
  - Computes actual worked duration from `currentWorkStartedAtMs`.
  - Saves a `WORK` session through `lib/session.ts` -> `POST /api/sessions`.
  - In fixed mode, switches to break and keeps running.
  - In flexible mode, computes a dynamic break and opens the break suggestion modal.
- On break completion:
  - Returns to work phase and idle status.

`TimerPanel` uses this engine plus `MusicPanel` and `BreakSuggestionModal`. The mounted `features/figma/TimerScreen.tsx` has a separate local timer implementation, including local mode/preset/phase timing, keyboard handling, break modal, completed-session count, and mode-switch confirmation.

## Break Logic

`lib/break.ts` contains the canonical dynamic break calculator:

- `break = workDurationSeconds * ratio`
- Ratio is clamped to `0.1` through `0.5`.
- Result is rounded to the nearest 30 seconds.
- Minimum break: 60 seconds.
- Maximum break: 20 minutes.

Note: the mounted Figma timer has its own `calcSuggestedBreak` helper using a simpler "worked minutes divided by 5" formula, capped differently. If break behavior must be consistent, reconcile this with `lib/break.ts`.

## Music / Audio

Music state is shared through Zustand. Audio generation lives in `features/music/hooks/use-focus-audio-engine.ts`.

Tracks:

- `deep-focus`: generated oscillator tones.
- `soft-rain`: generated looping noise buffer with lowpass filter.
- `alpha-pulse`: generated pulsing oscillator.

The audio engine:

- Uses Web Audio, including Safari `webkitAudioContext` fallback.
- Unlocks audio on first pointer/touch/key gesture.
- Fades gain in/out and cleans up source nodes.
- Reacts to `selectedTrackId`, `isMusicPlaying`, and `musicVolume`.

The Figma timer also calls this hook and maps its UI sound choices to the shared track IDs.

## Data Model

`prisma/schema.prisma` defines one model:

`Session`

- `id`: string cuid primary key
- `mode`: enum `FIXED | FLEXIBLE`
- `type`: enum `WORK | BREAK`, default `WORK`
- `durationSec`: integer
- `startedAt`: DateTime
- `endedAt`: optional DateTime
- `createdAt`: DateTime default `now()`

`prisma.config.ts` reads `DATABASE_URL` from the environment. `lib/prisma.ts` creates `PrismaClient` with `PrismaPg` adapter and uses this fallback only outside production:

`postgresql://postgres:postgres@localhost:5432/flexodoro?schema=public`

Important production guard: if `NODE_ENV === "production"` and `DATABASE_URL` is missing, app startup throws an explicit error and refuses localhost fallback. This prevents silent misconfiguration in hosted environments.

The Prisma client is cached on `global.prisma` outside production to avoid dev hot-reload connection churn.

## API Surface

`POST /api/sessions`

- File: `app/api/sessions/route.ts`
- Accepts `mode`, optional `type`, `durationSec`, `startedAt`, optional `endedAt`.
- Writes a `Session` through Prisma.
- Performs structured validation with explicit `400`/`422` responses for malformed JSON, invalid payload shape, enum/date/duration errors, unknown fields, and invalid `endedAt < startedAt` ranges.
- On Prisma known request errors, logs include Prisma `code` and `meta` to improve production diagnosis while still returning a generic `500`.

`GET /api/stats`

- File: `app/api/stats/route.ts`
- Reads only `WORK` sessions.
- Selects `durationSec` and `startedAt`.
- Returns `buildStatsSummary`.

`GET /api/insights`

- File: `app/api/insights/route.ts`
- Reads only `WORK` sessions.
- Selects `durationSec` and `startedAt`.
- Returns `buildInsightsSummary`.

`GET /api/health`

- File: `app/api/health/route.ts`
- Readiness-oriented health endpoint.
- Checks DB connectivity (`SELECT 1`) and `Session` schema readiness (`prisma.session.findFirst`).
- Returns `200` with `status: "ok"` when checks pass.
- Returns `503` with `status: "degraded"` and error details when checks fail.

## Stats Aggregation

`lib/stats.ts` exports:

- `buildStatsSummary`
- `buildInsightsSummary`
- `StatsSummary`
- `InsightsSummary`

`buildStatsSummary` returns:

- `totalTimeSec`
- `avgSessionSec`
- `sessionsCount`
- `dailyFocusTimeSec`
- `weeklyTrend` with 7 date buckets

Important detail: daily and weekly buckets use local date keys (`YYYY-MM-DD` from local date parts), and insights hour aggregation uses local `Date#getHours()`.

`buildInsightsSummary` returns:

- `bestFocusTime`
- `avgSessionLengthSec`
- `personalBestDay` (`date` + `totalDurationSec`, or `null` when no sessions exist)
- `hourlyFocusSec` (24-point hour distribution for `0..23`)

## Known Technical Observations

1. UI integration split:
   Mounted timer/stats screens are Figma-derived, while `TimerPanel` and `StatsDashboard` are more data-integrated but not the primary mounted pages.

2. Route/nav mismatch:
   `/app` redirects to `/`, but `AppLayout` marks timer active only when pathname is `/app`. The Timer nav links to `/`, so timer active styling may not show on the actual timer route.

3. Stats implementation status:
   `/app/stats` is Figma-mounted but API-backed (`/api/stats`, `/api/insights`) and includes loading/empty/error states. `StatsDashboard` still exists but is not the mounted route.

4. Break formula mismatch:
   Canonical `lib/break.ts` uses ratio/clamp/nearest-30s/min/max logic. Figma timer uses its own simpler formula. Future break-related work should pick one source of truth.

5. Persistence mismatch potential:
   Zustand persists preferences, but the mounted Figma timer stores its mode separately under `flexodoro.timer.mode`. Future mode/default behavior may need consolidation.

6. Session API validation:
   `/api/sessions` validates payload shape and fields, returning structured `400`/`422` errors before Prisma writes.

7. Timezone handling:
   Stats daily/weekly buckets and insights hours are local-time based. Stats UI parses local date keys safely to avoid UTC string parsing drift.

8. Break sessions are modeled but usually not analyzed:
   Schema supports `BREAK`, but stats/insights read only `WORK` sessions.

9. Design/prototype artifacts are intentionally present:
   `design/` is source/reference material and ignored by TypeScript/ESLint. Do not treat it as mounted runtime app code unless explicitly asked.

10. Production DB readiness dependency:
   Deployments require both a reachable `DATABASE_URL` and applied Prisma schema (`Session` table). `GET /api/health` now surfaces these failures directly (`degraded`) instead of only indicating liveness.
